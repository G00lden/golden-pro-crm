# الإصدار 1.2.0 — مركز التواصل الموثوق

تاريخ الإعداد: 2026-07-12

الحالة: جاهز برمجياً، غير منشور على الإنتاج حتى استعادة مفتاح SSH وإغلاق إعدادات المزود.

## ما تم تطويره

| التغيير | الربط مع الأقسام الأخرى | الأثر غير الملحوظ |
|---|---|---|
| توحيد رقم الهاتف في `shared/phone.ts` | العملاء، طلبات سلة، CRM Odoo الداخلي، المكالمات، واتساب، وبوابة SMS | البحث والمطابقة أصبحا يستخدمان رقماً دولياً واحداً. لم تُعد كتابة السجلات التاريخية؛ ما زالت المطابقة القديمة تعمل بآخر 9 أرقام لتجنب فقد الربط. |
| سجل `communication_events` | Webhook واتساب وحالات التسليم والرد الآلي | إعادة Meta لنفس الحدث لا تشغّل الرد أو المهمة مرتين. إذا مات العامل أثناء الحدث يمكن تحرير المطالبة وإعادة المحاولة. |
| طابور `communication_jobs` | المكالمة الفائتة ← واتساب العميل والموظف ← سجل المكالمة | الـWebhook يرجع بسرعة ولا ينتظر Meta. الرسائل تتأخر حتى ثانيتين عادة، وتُعاد بمحاولات متدرجة، وتنتهي بعد 30 دقيقة كي لا تصل رسالة مكالمة قديمة بعد إصلاح التوكن. |
| حالات إرسال أحادية الاتجاه | واتساب ← سجل الرسائل ← لوحة الإحصاءات | وصول `delivered/read` متأخراً لا يرجع الرسالة إلى حالة أقدم، وفشل قديم لا يمسح إثبات التسليم. |
| قوالب Cloud API مستقلة | الصيانة، الحجوزات، الفنيون، المكالمات الفائتة | لكل استخدام متغير بيئة مستقل وترتيب placeholders صريح. يلزم اعتماد القالب المقابل في Meta قبل الإنتاج. |
| مسار وارد موحد | WhatsApp Cloud/Web وSMS Gateway ← تأكيدات الصيانة واختيار قسم الاتصال | الرد `1` يُفسر بالطريقة نفسها من كل قناة. كلمات تأكيد الصيانة الحالية لها الأولوية حتى لا تتغير نتائج العملاء القدماء. |
| correlation مستقل للمكالمة | Unifonic ← `call_logs` ← القسم والموظف ← مهمة واتساب | اتصال العميل الجديد لا يكتب فوق اتصاله السابق. عند غياب معرف المزود تُربط خطوات المكالمة النشطة ضمن نافذة 10 دقائق فقط. |
| claim/lease لبوابة Android | صندوق SMS ← جوال الإرسال ← إقرار النجاح/الفشل | يقل التكرار عند وجود أكثر من ساحب. ما زال الضمان `at-least-once`: ضياع إقرار الجوال قد يعيد الرسالة بعد 60 ثانية، لذلك يجب أن يحفظ تطبيق الجوال معرف الرسالة محلياً. |
| فحص حي لـCloud API | حالة واتساب ← التذكيرات والمكالمات والبوابة البديلة | وجود التوكن لم يعد يعني «متصل». عند رفض Meta للتوكن تظهر الحالة خطأ وينتقل الإرسال المباشر إلى SMS بدلاً من نجاح وهمي. الفحص لا يرسل رسالة للعميل. |
| احترام تعطيل الرد الآلي | إعداد نظام المكالمات ← Webhooks | عند التعطيل يسمع المتصل رسالة توقف قصيرة، ولا تُنشأ سجلات/رسائل جديدة من أحداث الحالة. |
| لوحة مراقبة الطابور | شاشة واتساب ← وظائف المكالمات والحملات | تعرض المنتظر، المرسل، المحتاج لتدخل، والمنتهي. الظهور لا يعيد تشغيل الرسائل الفاشلة تلقائياً. |

## ما لم يُفعّل بعد

- توكن WhatsApp Cloud في الإنتاج مرفوض حالياً، لذلك لا توجد جاهزية فعلية للإرسال.
- قوالب `missed_call_customer` و`missed_call_agent` تحتاج أسماء قوالب Meta معتمدة في متغيرات الإنتاج.
- `WHATSAPP_APP_SECRET` و`WHATSAPP_WEBHOOK_SECRET` يحتاجان الضبط قبل قبول Webhook الإنتاجي.
- رقم Unifonic الرئيسي، رابط IVR، رابط حالات الاتصال، الأقسام والموظفون غير مجهزين في الإنتاج.
- محرك الحملات التسويقية الكامل—الموافقة، الإلغاء، الجمهور، الجدولة، حدود التكرار، الإيقاف والاستكمال—ليس جزءاً من 1.2.0 بعد. البنية الحالية هي الأساس الموثوق الذي سيستخدمه دون إرسال مباشر من Webhook.
- اسم `CRM Odoo` الحالي يعني مساحة CRM داخل SQLite؛ لا يوجد موصل Odoo خارجي حتى الآن.

## التحقق

- فحص TypeScript: ناجح.
- اختبارات الوحدة والترحيل: 66/66 ناجحة.
- بناء الإنتاج: ناجح.
- فحص المتصفح: شاشة واتساب والطابور سليمة على سطح المكتب وعرض جوال 390×844.
- محاكاة ترحيل لقطة الإنتاج: `user_version=10200` و`integrity_check=ok`.
- البيانات بعد المحاكاة: 254 عميلاً، 65 طلب متجر، 3 صفقات CRM، مهمة CRM واحدة، و33 رسالة واتساب.
- أنشأ الترحيل نسخة تلقائية قبل المخطط 10200 ولم يلمس لقطة المصدر.

## بوابة النشر

لا يُنشر 1.2.0 قبل:

1. إعادة مفتاح SSH إلى الجهاز والتحقق من الدخول غير التفاعلي.
2. إنشاء نسخة VPS ونسخة قاعدة بيانات قبل الترحيل.
3. تشغيل فحص التوكن والقوالب وWebhook من بيئة الإنتاج.
4. إبقاء الإرسال على `dry_run` أو allowlist حتى تمر مكالمة حقيقية واحدة دون تكرار.
5. التحقق من `/api/version` و`/api/health` وسلامة SQLite بعد النشر.
