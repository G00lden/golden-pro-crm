# الإصدار 1.3.0 — الحملات الآمنة

تاريخ الإعداد: 2026-07-12

الحالة: مكتمل برمجياً ومرفوع للاختبار بعد التزام Git؛ النشر على VPS ينتظر إضافة مفتاح SSH من جلسة Contabo.

## ما تم بناؤه

| الميزة | الربط مع الأقسام | الأثر غير الملحوظ |
|---|---|---|
| موافقة تسويق صريحة لكل رقم وقناة | العملاء، عملاء سلة، CRM الداخلي، واتساب وSMS | عدم وجود سجل `granted` يعني أن العميل غير مؤهل؛ لا تُفترض الموافقة من مجرد وجود الرقم أو طلب سابق. |
| قائمة إلغاء مستقلة | الوارد من Cloud API وWhatsApp Web وSMS | كلمات `إلغاء` و`STOP` الدقيقة تسحب الموافقة وتضيف suppression مرة واحدة. عبارة مثل «إلغاء موعد الصيانة» لا تُفسر كإلغاء تسويقي. |
| فحص مزدوج للموافقة | الجمهور عند المعاينة + العامل لحظة الإرسال | إذا ألغى العميل بعد الجدولة وقبل دوره، تُحظر الرسالة. قد تمر رسالة واحدة فقط إذا أصبح طلب الإلغاء بعد بدء طلب Meta فعلياً؛ لا يمكن سحب طلب شبكي أصبح قيد التنفيذ. |
| كيان حملة وحالات تشغيل | شاشة الحملات، الطابور، سجل واتساب | الحالات: مسودة، مجدولة، تعمل، متوقفة، مكتملة، ملغاة. الإلغاء يمنع الوظائف المعلقة ولا يرجع الرسائل التي قُبلت سابقاً من المزود. |
| شرائح الجمهور | جدول العملاء المشترك مع سلة وCRM | الفلاتر الحالية: كل العملاء، المدينة، المصدر، أو معرفات عملاء محددة. التطابق بالهاتف الموحد يمنع تكرار العميل نفسه داخل الحملة. |
| حد تكرار | نتائج الحملات السابقة | العميل الذي استلم حملة خلال المدة المحددة يُستبعد حتى لو بقي موافقاً. هذا يقلل الجمهور عمداً ويحمي جودة الرقم. |
| تحديد معدل الإرسال | `communication_jobs.available_at` | لا تُرسل الدفعة مرة واحدة؛ توزع حسب عدد الرسائل في الدقيقة. تغيير الحد بعد إنشاء الوظائف لا يعيد جدولة الوظائف الموجودة في هذا الإصدار. |
| إيقاف واستكمال | حالة الحملة + العامل | الإيقاف يؤجل الوظيفة 60 ثانية دون احتساب محاولة فاشلة. وظيفة تجاوزت بوابة الفحص وبدأ طلبها الشبكي قد تكتمل أثناء الضغط على الإيقاف. |
| ربط إيصالات Meta | `whatsapp_messages` + مستلمي الحملة | حالات `sent/delivered/read/failed` تظهر في أرقام الحملة ولا تتراجع عند وصول callbacks بترتيب مختلف. |
| بوابة تشغيل إنتاجية | حالة Cloud API + outbound safety + قالب Meta | إنشاء المسودات والمعاينة متاحان دائماً، لكن التشغيل يتطلب Cloud API متحققاً ووضع `production` مع اعتماد الإطلاق وقالب `general_reminder` معتمد. |
| واجهة إدارية مستقلة | صلاحيات admin/manager في CRM | المستخدم العادي لا يرى الصفحة. إضافة الموافقة تحتاج دليلاً نصياً، ورفع حظر سابق يحتاج اختياراً صريحاً. |

## ما لم يدخل هذا الإصدار

- لا يوجد A/B testing أو اختيار نسخة فائزة تلقائياً.
- لا يوجد استيراد موافقات جماعي؛ هذا مقصود حتى يُحدد مصدر الإثبات وصيغة الملف.
- لا يوجد موصل Odoo خارجي؛ الربط الحالي يمر عبر جدول عملاء CRM الداخلي.
- لا تُستخدم قوالب الصيانة والمكالمات كقوالب حملات جماعية. يجب إنشاء قوالب تسويق مستقلة قبل توسيع القائمة.
- لا يوجد تعديل لمسودة بعد إنشائها؛ يمكن إلغاؤها وإنشاء مسودة جديدة مع سجل تدقيق أوضح.

## التحقق المنفذ

- اختبارات الوحدة: 75 ناجحة.
- اختبارات التكامل: 15/15 ناجحة.
- المسار الذهبي: 10/10 ناجح.
- تدقيق المصدر الأمني: صفر تحذيرات.
- بناء الإنتاج: ناجح.
- فحص المتصفح: سطح المكتب والجوال ناجحان.
- اختبار واجهة فعلي: إنشاء مسودة ومعاينتها دون إرسال، والجمهور غير الموافق بقي صفراً وزر التشغيل معطلاً.
- محاكاة ترحيل لقطة الإنتاج: `integrity_check=ok` و`user_version=10300`.
- بقي بعد الترحيل: 254 عميلاً، 65 طلب متجر، 7 عروض، 9 فواتير، 3 صفقات، و33 رسالة واتساب.
- الترحيل أنشأ نسخة `pre-schema-10300` تلقائياً ولم يغير لقطة المصدر.

## بوابة النشر

حتى بعد نشر الكود ستبقى الحملات مقفلة في الإنتاج الحالي لأن فحص WhatsApp Cloud كان يرفض التوكن. يلزم قبل أول إرسال:

1. استبدال التوكن والتحقق الحي من الاتصال.
2. اعتماد قالب Meta بمتغير نص واحد وربطه بـ`WHATSAPP_CLOUD_TEMPLATE_NAME`.
3. ضبط أسرار Webhook والتحقق من HMAC.
4. إدخال الموافقات الموثقة فقط.
5. تجربة allowlist منفصلة قبل تغيير وضع التشغيل إلى إنتاج معتمد.
