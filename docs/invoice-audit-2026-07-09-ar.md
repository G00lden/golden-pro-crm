# تقرير تقني — قسم الفواتير (golden-pro-crm)

**التاريخ:** 2026-07-09 · **المُعِدّ:** Claude Code · **النطاق:** الفواتير الضريبية (زاتكا) فقط — لم تُلمَس ملفات المتجر (ملك هيرميز).

هذا تقرير **تحليلي**. نُفِّذ الآن إصلاح واحد فقط (تصغير اللوجو). بقية البنود موثّقة هنا وتنتظر موافقة المالك قبل التنفيذ، وفق قراره «تقرير أولًا».

## خريطة المسارات (مهمّة لفهم كل بند)
منطق الفاتورة مكرّر عبر **ثلاثة مسارات كتابة**، لكلٍّ نسخته من الحساب/الترقيم/`paid_at`:
1. **محلي (وضع المالك الفعلي):** `src/api.ts` — `localInvoicePayload` (~1701)، فرع `createInvoice` المحلي (~1761)، `updateInvoice` المحلي (~1786)، `setInvoiceStatus` المحلي (~1830). يعمل في المتصفح/localStorage.
2. **الخادم (SQLite خلف نفق Cloudflare عبر `DATA_PROVIDER=sqlite`، وSupabase):** مسارات `server/crmApi.ts` → محوّل `adminDb` → `server/sqliteFirestoreAdapter.ts` → `server/db.ts`.
3. **Firestore (SDK العميل):** الفروع غير المحلية في `src/api.ts`.

> أي إصلاح لمنطق الفاتورة يجب أن يُطبَّق في المسارات الثلاثة معًا وإلا تباعدت النتائج. هذا هو جوهر التوصية بمصدر حساب واحد (البند 6).

## عالٍ (HIGH)

### 1) تعارض بنود الجدول مع الإجماليات وQR عند وجود خصم — **مُعلَّم: للمحاسب، لا تغيير الآن**
- **المكان:** `invoiceLineAmounts` (`server/crmApi.ts:1007`، `src/pages/Invoices.tsx:727`) مقابل `invoiceTotals` (`src/api.ts:1656`، `server/crmApi.ts:963`).
- **الوصف:** حساب مبالغ **البنود** يتجاهل الخصم تمامًا، بينما حساب **الإجماليات** يطرح الخصم قبل الضريبة. فعند `discount > 0`:
  - `Σ صافي البنود = subtotal` لكن `total_without_vat = subtotal − discount`.
  - `Σ ضريبة البنود = subtotal·rate` لكن `vat_amount = (subtotal − discount)·rate`.
  - جدول البنود المطبوع، وصندوق الإجماليات، ورمز QR زاتكا (الذي يرمّز `total_with_vat`/`vat_amount` من الترويسة) — **كلها تتناقض** كلما وُجد خصم. عند `discount = 0` تتطابق.
- **الأثر:** فاتورة ضريبية بنودها لا تُجمَع إلى إجمالياتها/QR = خلل امتثال محتمل.
- **الحل المقترح (بانتظار المحاسب):** دالة `distributeDiscount(items, discount, vatPercent)` توزّع الخصم على البنود بالتناسب مع صافي كل بند، ثم تحسب صافي/ضريبة/إجمالي كل بند بعد الخصم، مع **تسوية القرش الأخير** (largest-remainder) بحيث `Σ` البنود = الترويسة = QR بالضبط. يحتاج إقرار المحاسب: أن الخصم غير شامل للضريبة (كما هو اليوم)، وأن التقريب نصف‑لأعلى مع وضع القرش المتبقّي على آخر بند.

### 2) عدم اتساق ختم `paid_at`
- **المكان:** الإنشاء `POST /api/invoices` (`server/crmApi.ts:1231`) يثبّت `paid_at: null` حتى لو أُنشئت بحالة `paid`؛ التحديث `PUT /api/invoices/:id` (`~1268–1288`) يكتب `status` لكن لا يلمس `paid_at` أبدًا؛ ومسار Firestore في `updateInvoice` (`src/api.ts:~1802`) كذلك. فقط `POST /:id/status` (`~1309`) يختم الوقت بشكل صحيح.
- **الأثر:** يمكن أن تصبح الفاتورة `paid` بلا `paid_at` (عبر الإنشاء أو PUT) → ضياع وقت الدفع، وعدم اتساق حسب المسار المستخدم.
- **الحل:** قاعدة واحدة `paidAtFor(nextStatus, existingPaidAt, now)`: إن كانت الحالة الجديدة `paid` → `existingPaidAt || now`، وإلا احفظ `existingPaidAt`. تُطبَّق في الإنشاء وPUT والحالة عبر المسارات الثلاثة.

## متوسط (MEDIUM)

### 3) لا توجد آلة حالات (state machine) للفاتورة
- **المكان:** `POST /api/invoices/:id/status` (`server/crmApi.ts:1294`) يتحقّق فقط أن القيمة ضمن `{draft, issued, sent, paid, cancelled, refunded}` — أي انتقال من أي حالة إلى أي حالة مسموح.
- **الأثر:** يمكن `paid→draft`، `cancelled→paid`، `refunded→issued` — إحياء فاتورة ملغاة/مستردة أو عكس دفعة. ثغرة متانة/تدقيق لوثيقة ضريبية. (وأيضًا `paid→cancelled` يُبقي `paid_at`، ثم `cancelled→paid` يُبقي القديم بدل إعادة الختم.)
- **الحل:** خريطة انتقالات مسموحة:
  - `draft → issued | sent | cancelled`
  - `issued | sent → paid | cancelled`
  - `paid → refunded`
  - `cancelled | refunded → (نهائية)`
  
  مع السماح بالانتقال إلى نفس الحالة (لا عملية). تُفرَض في مسار الحالة **وPUT** برفض 409 ورسالة عربية، وتعطيل الأزرار غير المنطقية في الواجهة `Invoices.tsx`.

### 4) سباق رقم الفاتورة (قد يُنتج تكرارًا أو خطأ 500)
- **المكان:** الإنشاء يقرأ `listOwned(..., 10000)` ثم `nextSequence(all, ...)` (`server/crmApi.ts:~1205–1210`) — قراءة ثم كتابة مع `await` بينهما، بلا قفل.
- **الأثر:**
  - **Firestore:** لا قيد تفرّد → إنشاءان متزامنان قد يكتبان نفس `invoice_number` (خرق تفرّد زاتكا).
  - **SQLite:** يوجد `UNIQUE INDEX idx_invoices_owner_number` (`server/db.ts:454/769`) فيرمي `SQLITE_CONSTRAINT` على الخاسر → **500 غير مُعالَج**.
  - سقف `10000` قد يقصّ الحد الأقصى لمالك تجاوز 10k فاتورة فيتصادم/يتراجع الرقم.
- **الحل:** عدّاد ذرّي لكل مالك:
  - **SQLite:** جدول `invoice_counters(owner_uid PRIMARY KEY, seq)` مع `INSERT ... ON CONFLICT(owner_uid) DO UPDATE SET seq=seq+1 RETURNING seq` داخل `db.transaction` (يُبذَّر من `MAX(sequence)` الحالي أول مرة).
  - **Firestore:** `runTransaction` على `invoice_counters/{uid}` مع `increment(1)`.
  - غلاف **إعادة محاولة (≤3)** حول `createOwned` يلتقط خطأ التفرّد ويعيد التخصيص — فلا 500 ولا تكرار. يُشحَن كـPR مستقل (يمسّ المحوّل + جدولًا جديدًا).

### 5) اختلاف التقريب بين العميل والخادم (يغذّي QR زاتكا)
- **المكان:** الخادم `invoiceTotals` يعيد `vat_amount` **غير مقرّب** (`server/crmApi.ts:978`، يُقرّب `total_with_vat` فقط في :980)، بينما العميل يقرّب `vat_amount` لخانتين (`src/api.ts:1670`). كما يختلف **ترتيب** التقريب: العميل `total = round(withoutVat + withoutVat·rate)` مقابل الخادم `total = round(afterDiscount + round(afterDiscount·rate))`.
- **الأثر:** الفاتورة نفسها قد تحمل قيمة ضريبة/إجمالي تختلف بقرش حسب المسار (محلي/Firestore مقابل الخادم)، والقيمة المخزّنة تغذّي QR زاتكا.
- **الحل:** ترتيب تقريب واحد قانوني في الوحدة المشتركة: قرّب الضريبة أولًا، ثم `total = withoutVat + roundedVat`، ثم قرّب.

## منخفض/هيكلي (LOW / structural)

### 6) حساب الضريبة مكرّر في ~4 أماكن — أساس «الترابط التقني» المطلوب
- **المكان:** `invoiceTotals`/`invoiceLineAmounts` مستقلة في: `src/api.ts:1656`، `src/pages/Invoices.tsx:727`، `server/crmApi.ts:963`+`:1007`، إضافةً إلى قالب HTML رابع في `publicInvoiceHtml` (`server/crmApi.ts:1022`). أي تغيير في قاعدة الضريبة يجب تعديله في كلها.
- **الحل:** مصدر واحد `src/shared/invoiceMath.ts` (TS نقي، بلا استيرادات خارجية — فقط `TextEncoder`/`btoa` المتوفّران في Node 18+ والمتصفح) يُصدِّر: `resolveVatPercent`, `roundMoney`, `invoiceTotals`, `invoiceLineAmounts`, `distributeDiscount`, `paidAtFor`, `INVOICE_TRANSITIONS`/`canTransition`, `zatcaTlvBase64`. يستورده العميل (Vite) والخادم (tsx) بلا مشاكل تجميع. الإصلاح الأدنى خطرًا: اجعل الدوال الحالية **تفوّض** للوحدة بدل حذف مواقع الاستدعاء دفعة واحدة.

### 7) طول TLV ببايت واحد يفسد QR للأسماء الطويلة
- **المكان:** `generateTLV` (`server/crmApi.ts:844`) و`generateInvoiceQRCode` (`src/api.ts:~1954`) يرمّزان طول كل وسم في بايت واحد (`len.toString(16).padStart(2,"0")`).
- **الأثر:** حقل ≥256 بايت (اسم بائع عربي طويل بترميز UTF-8) يفيض فيُنتج هكس خاطئًا ويُفسد سلسلة TLV/QR. حالة حديّة لكنها حقيقية.
- **الحل:** ترميز دفاعي للطول (حراسة/رفض >255 بدل الإفساد الصامت) مُوحَّد داخل `zatcaTlvBase64` في الوحدة المشتركة.

### 8) سجل/جوال البائع + عمود qr_code — **قرار المالك: يبقيان ثابتين**
- **المكان:** `seller_cr` (`7016449519`) و`seller_phone` (`+966533971168`) مطبوعان في `publicInvoiceHtml` (`server/crmApi.ts:1026–1027`) و`Invoices.tsx:760`، لكن لا حقل في الإعدادات ولا عمود في قاعدة البيانات → غير قابلين للضبط. وعمود `qr_code` (`server/db.ts:448`) موجود لكنه **لا يُكتَب أبدًا** (QR يُعاد توليده عند كل عرض).
- **القرار:** يبقيان ثابتين كما هما الآن — موثّق فقط، بلا تغيير.

### 9) تناقضات عبر المستندات (للعلم)
- **تنسيق الأرقام:** `public/quotation-template.html` يستخدم `toLocaleString('ar-SA')` (أرقام هندية، بلا كسور ثابتة) مقابل `toLocaleString('en-US', {minimumFractionDigits:2})` في الفاتورة — أسلوب أرقام وكسور غير متّسق بين المستندين.
- **هوية البائع:** قالب العرض ثابت على "Golden Pro International / جولدن برو" بينما الفاتورة ثابتة على "BreeXe Pro / بريكس برو".
- **الرقم الضريبي التجريبي:** `demoInvoice` (`src/api.ts:~2738`) يستخدم `300000000000003` بينما `defaultSettings` (`~655`) يستخدم `313049114100003`.

## استراتيجية التحقق عند التنفيذ (لاحقًا)
- **الخصم:** إنشاء فاتورة `discount>0` بأصناف مختلطة، والتأكد `Σ round(line.net)=total_without_vat`, `Σ line.vat=vat_amount`, `Σ line.gross=total_with_vat`، وفكّ QR والتأكد أن الوسمين 4/5 = ترويسة الإجماليات.
- **التقريب:** تمرير تشكيلات عشوائية عبر عميل/خادم والتأكد من تطابق `vat_amount`/`total_with_vat` للقرش.
- **paid_at:** إنشاء بحالة `paid` → مختوم؛ PUT مسودة→مدفوعة → مختوم؛ PUT مدفوعة→مدفوعة بأصناف جديدة → غير متغيّر.
- **آلة الحالات:** جدولة كل (من،إلى) والتأكد 409 على غير المنطقي و200 على المنطقي، وتعطيل الأزرار.
- **الترقيم:** 20 إنشاء متوازٍ → 20 رقمًا متمايزًا، صفر 500، بلا إعادة استخدام بعد الحذف.
- بوابة كل PR: `npm run lint` + `npm run build` + `npm run test:smoke`.
