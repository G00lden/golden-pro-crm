# مهمة هيرميز — تدقيق وتقوية تكامل سلة (Salla)

> صادرة من: Claude Code · التاريخ: 2026-07-06 · الحالة: مفتوحة
> تنسيق: Claude Code يدقّق مسار الرسائل/الواتساب بالتوازي — **لا تلمس** `server/whatsapp*.ts`، `reminderEngine.ts`، `ivrEngine.ts`، `routes-telephony.ts`، `maintenanceLifecycle.ts` (ملكية Claude Code مؤقتًا). أنت خذ ملفات سلة/المتجر فقط.

## الخلفية
انتهى تدقيق أخطاء شامل (`docs/bug-audit-2026-07-06.md`) وأُصلحت 49 مشكلة. لكن ظهرت مشاكل حقيقية أثناء التشغيل الفعلي لم يلتقطها التدقيق. من صورة رسالة واتساب فعلية لطلبات/متابعات عملاء من سلة ظهر:
- **أرقام بمقدّمات خاطئة**: `+90 552 033 25 71` (تركيا)، `+1 (866) 498-8885` (أمريكا) — بدل التطبيع السعودي `+966`. يبدو أن أرقام عملاء سلة تُخزَّن/تُعرض بصيغة دولية خام بدون تطبيع KSA.
- **اسم عميل مشوّه**: «عميلamribrahim49» — كلمة «عميل» ملتصقة باسم مستخدم (username) بدل الاسم الحقيقي؛ غالبًا عميل سلة بلا اسم فيرجع الـ handle، مع فقدان مسافة في الدمج.

## المطلوب (تدقيق + تقوية)
1. **تطبيع الهاتف لعملاء سلة**: تأكّد أن كل رقم من سلة يمر عبر `normalizePhone` بصيغة KSA (`966...`) عند **التخزين وعند العرض وعند الإرسال**. الأرقام الدولية غير السعودية (`+90`، `+1`): قرّر سلوكًا واضحًا — إمّا الحفاظ عليها كما هي بصيغة E.164 صحيحة، أو وسمها، لكن **لا** تُلصق بادئة 966 على رقم دولي فعلي. تتبّع من أين جاءت `+90/+1` في الصورة (بيانات فعلية أم خطأ تطبيع؟).
2. **اسم العميل**: أصلح دمج «عميل» + الاسم (مسافة مفقودة)، ووفّر fallback لائق عندما يكون اسم عميل سلة فارغًا/username بدل اسم حقيقي.
3. **الأسعار**: تأكّد أن إصلاح `numericValue` (PR #22 — النص الفارغ يرجع undefined بدل 0) يُستهلك صح في كل مسارات استيراد أسعار سلة، وما فيه مكان ثاني يصفّر السعر.
4. **مخطط store_orders**: بعد PR #20 (الأعمدة الـ12 أُضيفت)، تحقّق أن الاستيراد على قاعدة جديدة يكتب كل الأعمدة (customer_name/phone/city/product_*/order_status/installation_status…) بلا أخطاء.
5. **أمان الـ webhook + التوكن**: راجع HMAC على rawBody، منع التكرار (dedup keys)، وتجديد التوكن — أبلغ عن أي ثغرة.
6. **أي أخطاء تشغيلية أخرى** في مسار سلة (طلبات مكررة، حالات مفقودة، تعطّل صامت).
